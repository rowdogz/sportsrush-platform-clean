# SportsRush 2.0 — Database Schema Notes

Reference document for the D1 (SQLite) schema. The migration files are the source
of truth — this document provides rationale and cross-cutting design decisions.

## Applied migrations

| #    | File                          | What it does                                  |
| ---- | ----------------------------- | --------------------------------------------- |
| 0001 | `0001_foundation.sql`         | Baseline — enables `PRAGMA foreign_keys = ON` |
| 0002 | `0002_auth_schema.sql`        | Identity & Auth domain tables                 |
| 0004 | `0004_admin_audit_events.sql` | Admin/domain audit event trail                |

## Cross-cutting conventions

### Primary keys

All tables use **UUID v4 TEXT** primary keys generated by `crypto.randomUUID()`
in the application. There is no UUID column type in SQLite — TEXT is used.

### Timestamps

All timestamp columns are **TEXT in ISO 8601 UTC** format, e.g.:

```
2025-03-15T18:00:00.000Z
```

This matches the `Timestamp` branded type in `packages/types/src/common.ts`.
ISO 8601 strings are lexicographically sortable, so `WHERE expires_at < ?` comparisons work correctly.

SQL default: `DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now'))`

`updated_at` is managed automatically by **AFTER UPDATE triggers** (see Triggers section below).
Application UPDATE statements do not need to include `updated_at` — the trigger sets it.
Application code MAY supply an explicit `updated_at` (e.g. data migrations, test fixtures);
the trigger's `WHEN NEW.updated_at IS OLD.updated_at` guard ensures it will not be overridden.

### Foreign key enforcement

`PRAGMA foreign_keys = ON` is a session-level SQLite setting (not persisted).
It is applied in two places:

1. Each migration file runs it for the migration session.
2. `createDbClient()` in `src/lib/db.ts` runs it on every request before any query.

### Boolean fields

Stored as `INTEGER NOT NULL DEFAULT 0` (SQLite has no BOOLEAN type).
`0` = false, `1` = true. The application maps these to TypeScript booleans.

### Immutable tables

`auth_audit_log` and `audit_events` are **append-only**.
They have no `updated_at` column. Rows are never updated or deleted.

## Table reference (PR-06)

### `users`

Core identity record. One row per human user.

| Column                          | Type        | Notes                                                 |
| ------------------------------- | ----------- | ----------------------------------------------------- |
| `id`                            | TEXT PK     | UUID v4                                               |
| `email`                         | TEXT        | Original case, display only                           |
| `email_normalized`              | TEXT UNIQUE | `email.trim().toLowerCase()` — used for all lookups   |
| `email_verified_at`             | TEXT        | ISO 8601; null until email verified                   |
| `password_hash`                 | TEXT        | Null until set; format `$pbkdf2-sha256$...`           |
| `role`                          | TEXT        | `'user' \| 'admin' \| 'superadmin'`; CHECK constraint |
| `is_active`                     | INTEGER     | 0 = suspended; 1 = active                             |
| `is_legacy_migration`           | INTEGER     | 1 for users imported from WordPress                   |
| `legacy_wp_user_id`             | INTEGER     | Original WP numeric user ID; null for native          |
| `legacy_migration_completed_at` | TEXT        | ISO 8601; null until migration flow complete          |
| `created_at`                    | TEXT        | ISO 8601                                              |
| `updated_at`                    | TEXT        | ISO 8601; set automatically by `trg_users_updated_at` |

**Legacy migration flow:** A user with `is_legacy_migration = 1` and `password_hash = NULL`
must complete either a magic link login or a password reset before gaining full access.
`legacy_migration_completed_at` is set at that point. WordPress password hashes
(`$P$...`) are never stored — see `packages/auth/src/password.ts` (`isLegacyPhpassHash`).

### `user_profiles`

Display information. 1-to-1 with `users`. Created atomically with the `users` row.

| Column         | Type               | Notes                                                         |
| -------------- | ------------------ | ------------------------------------------------------------- |
| `user_id`      | TEXT PK → users.id | CASCADE on delete                                             |
| `display_name` | TEXT               | Shown on leaderboards; max 3 changes per 24h (app-enforced)   |
| `avatar_url`   | TEXT               | Null until set                                                |
| `timezone`     | TEXT               | IANA string, e.g. `Europe/London`; validated by app           |
| `created_at`   | TEXT               | ISO 8601                                                      |
| `updated_at`   | TEXT               | ISO 8601; set automatically by `trg_user_profiles_updated_at` |

### `user_sessions`

Refresh token tracking. One row per active device/browser session.

| Column               | Type            | Notes                              |
| -------------------- | --------------- | ---------------------------------- |
| `id`                 | TEXT PK         | UUID v4                            |
| `user_id`            | TEXT → users.id | CASCADE on delete                  |
| `refresh_token_hash` | TEXT UNIQUE     | SHA-256 of the raw refresh token   |
| `user_agent`         | TEXT            | For session management UI          |
| `ip_address`         | TEXT            |                                    |
| `last_used_at`       | TEXT            | Updated on each token refresh      |
| `expires_at`         | TEXT            | ISO 8601; typically now + 30 days  |
| `revoked_at`         | TEXT            | ISO 8601; null for active sessions |

A session is valid when: `revoked_at IS NULL AND expires_at > strftime(..., 'now')`.

### `magic_links`

Passwordless login. Single-use, 15-minute TTL.

| Column             | Type        | Notes                                      |
| ------------------ | ----------- | ------------------------------------------ |
| `token_hash`       | TEXT UNIQUE | SHA-256 of URL-safe random token           |
| `email_normalized` | TEXT        | Email the link was sent to                 |
| `expires_at`       | TEXT        | ISO 8601; now + 15 minutes                 |
| `used_at`          | TEXT        | ISO 8601; null until consumed — single-use |

### `oauth_accounts`

OAuth 2.0 provider linkage. One row per (user, provider) pair.

| Column             | Type | Notes                                                   |
| ------------------ | ---- | ------------------------------------------------------- |
| `provider`         | TEXT | e.g. `'google'`, `'facebook'` — no CHECK, app validates |
| `provider_user_id` | TEXT | Stable ID from provider (not email)                     |
| `email`            | TEXT | From provider; informational only                       |

UNIQUE on `(provider, provider_user_id)`.

### `password_reset_tokens`

Single-use, 15-minute TTL.

| Column       | Type        | Notes                                          |
| ------------ | ----------- | ---------------------------------------------- |
| `token_hash` | TEXT UNIQUE | SHA-256 of URL-safe random token               |
| `expires_at` | TEXT        | ISO 8601; now + 15 minutes                     |
| `used_at`    | TEXT        | ISO 8601; null until password change committed |

### `auth_audit_log`

Append-only authentication and security events. No `updated_at`.

| Column       | Type            | Notes                                      |
| ------------ | --------------- | ------------------------------------------ |
| `user_id`    | TEXT → users.id | SET NULL on delete (preserves audit trail) |
| `event_type` | TEXT            | See event type list in migration file      |
| `ip_address` | TEXT            |                                            |
| `user_agent` | TEXT            |                                            |
| `metadata`   | TEXT            | JSON blob; schema is per-event-type        |

### `audit_events`

Append-only admin/domain mutation events. No `updated_at`.

| Column            | Type            | Notes                                        |
| ----------------- | --------------- | -------------------------------------------- |
| `actor_user_id`   | TEXT → users.id | SET NULL on delete (preserves audit trail)   |
| `action`          | TEXT            | Stable action key, e.g. `competition.update` |

| `target_id`       | TEXT            | Entity id when available                     |
| `before_metadata` | TEXT            | JSON-safe state before mutation              |

## Email normalisation

Two columns are maintained on `users`:

```
email            ← original case ("Alice@Example.COM")   — display only
email_normalized ← lowercased + trimmed ("alice@example.com") — all lookups
```

The application normalises before every INSERT and before every lookup:

```typescript
const normalised = email.trim().toLowerCase();
```

The UNIQUE constraint is on `email_normalized`. Attempting to register
`Alice@Example.COM` when `alice@example.com` already exists will fail with
a unique constraint violation, which the auth service maps to a `ConflictError`.

## Token security

All security tokens follow this protocol:

1. Application generates a cryptographically random URL-safe token (e.g. 32 bytes → 43 base64url chars)
2. Application sends the raw token to the user (email link, HTTP-only cookie, etc.)
3. Application hashes the token with SHA-256 and stores the hash
4. On verification: received token is hashed and compared to stored hash
5. `used_at` is set atomically with the action it authorises (single-use enforcement)

This means a database breach exposes only hashes — raw tokens are never stored.

## Indexes

| Index                               | Table                 | Purpose                                       |
| ----------------------------------- | --------------------- | --------------------------------------------- |
| `idx_users_role`                    | users                 | Admin role-based queries                      |
| `idx_users_legacy_wp_user_id`       | users                 | Import deduplication; partial (non-null only) |
| `idx_user_sessions_user_id`         | user_sessions         | Logout-all, session list                      |
| `idx_user_sessions_expires_at`      | user_sessions         | Expiry cleanup job                            |
| `idx_magic_links_user_id`           | magic_links           | Pending links per user                        |
| `idx_magic_links_expires_at`        | magic_links           | Expiry cleanup job                            |
| `idx_oauth_accounts_user_id`        | oauth_accounts        | All linked providers for a user               |
| `idx_password_reset_tokens_user_id` | password_reset_tokens | Pending resets per user                       |
| `idx_auth_audit_log_user_id`        | auth_audit_log        | User audit trail                              |
| `idx_auth_audit_log_event_type`     | auth_audit_log        | Event type filtering                          |
| `idx_auth_audit_log_created_at`     | auth_audit_log        | Time-range queries                            |

UNIQUE constraints automatically create indexes on:
`users.email_normalized`, `user_sessions.refresh_token_hash`, `magic_links.token_hash`,
`oauth_accounts.(provider, provider_user_id)`, `password_reset_tokens.token_hash`.
These are not duplicated in the explicit index list.
